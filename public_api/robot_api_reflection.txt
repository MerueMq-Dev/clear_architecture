В этот раз мы решили задачу заметно по-разному, и сравнение получается
содержательное. У эталонного решения есть один большой метод activate_cleaner,
внутри которого живёт уже знакомая цепочка if/elif для разбора команд, плюс
несколько геттеров get_x, get_y, get_angle, get_state для доступа к состоянию.
Я пошёл другим путём: вынес каждое действие робота в отдельный метод move,
turn, set_mode, start, stop, а разбор команд оставил в чистом ядре через
функцию run_script. Если сравнивать стороны, мне кажется, у моего решения
логика действий разделена честнее. Эталонная activate_cleaner делает сразу две вещи:
разбирает строковые команды и вызывает действия робота. Это тот же самый
недочёт, который я в прошлый раз сделал у себя, только наоборот: там я
слишком дробил, здесь эталон, наоборот, всё слепил в один метод. У меня же
API оказался более общим: его можно использовать и через отдельные методы
напрямую, и через run_script для пакета команд. По сути я предоставил два
способа взаимодействия вместо одного.
А вот по доступу к состоянию у эталона решение лучше. Геттеры get_x, get_y,
get_angle, get_state — это явный интерфейс чтения, который отделяет
внутреннее устройство от внешнего и позволяет позже менять реализацию,
не ломая клиентский код. Я обошёлся без них, и зря — для API-класса такие
точки доступа как раз уместны. Возможно, в питоническом стиле их стоило бы
оформить через @property, чтобы снаружи это читалось как обычные поля,
но сама идея отдельных методов доступа правильная.
Из того, что у эталона лучше: transfer_to_cleaner определена прямо в API,
а не дёргается из ядра. Я взял её из pure_robot, и это смешало слои —
интерфейс с пользователем по-хорошему должен сам решать, как выводить
сообщения, а не таскать функцию из чистого модуля. Это та граница,
которую я в этот раз провёл нечётко.
